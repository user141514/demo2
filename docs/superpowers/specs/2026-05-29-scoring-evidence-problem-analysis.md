# 评分系统证据质量问题分析

**日期**: 2026-05-29  
**系统**: 行动学习评分系统 (Flask + SQLite)  
**范围**: 证据 (evidence) 生成的两个路径 — LLM 路径和启发式路径  
**用户反馈**: "评分依据都是偏门信息"

---

## 一、当前架构概览

评分系统有两个并行路径，均由 `scoring_app/scoring.py::score_submission()` 统一入口调度：

```
score_submission()
  ├── 路径 A: LLM 路径 (live_scoring.py)
  │   └── 调用 OpenAI 兼容 API → JSON 解析 → _normalize_dimensions()
  └── 路径 B: 启发式路径 (scoring.py)
      └── _build_heuristic_dimensions()
          ├── _score_dimension()      → 基于关键词覆盖率的分数计算
          ├── _build_evidence()       → 第一匹配句子提取
          └── _build_comment()        → 基于分数等级的模板化评语
```

---

## 二、具体问题分析

### 问题 1: 第一匹配偏见 — 关键词触发即证据

**涉及代码**: `scoring.py` 第 213-224 行

```python
def _build_evidence(text, keywords):
    sentences = _split_sentences(text)
    for keyword in keywords:                # 外层: 按维度关键词列表顺序
        for sentence in sentences:          # 内层: 按文档顺序
            if keyword in sentence and not looks_like_garbled_text(sentence):
                return _limit(sentence, 80)  # 立即返回第一个匹配!
    # 兜底: 取第一个非乱码句子
    ...
```

**核心缺陷**:
- 只要句子包含**任意一个**关键词，就立即成为证据
- 完全不评估该句子与维度**核心含义**的匹配程度
- 不比较多个候选句子之间的质量差异
- 关键词顺序决定了搜索优先级，与语义无关

**具体场景 — 维度 5 "反思深刻性与真诚度" (关键词: 反思, 复盘, 不足, 改进, 认知, 迭代, 收获)**

假设文档内容:
```
"...我们通过流程改进提升了交付效率。..."
"...我在复盘中深刻反思了自己的认知局限，意识到过去对团队管理的理解停留在表面。..."
```

算法首先遍历关键词 "反思", 在文档中未找到; 然后遍历 "复盘", 在第二句中匹配。但如果关键词顺序是 `[改进, 反思, 复盘, ...]`（当前代码中 `rules.py` 第 72-73 行的排列），则算法**先**匹配到第一句中的 "改进"，立即返回 "我们通过流程改进提升了交付效率" 作为该维度的 evidence。

该句子与 "反思深刻性" 完全无关，它描述的是流程改进，不是自我反思。这直接解释了用户反馈的"偏门信息"。

### 问题 2: 跨维度关键词碰撞 — 同一证据用于不同含义

**涉及代码**: `rules.py` 各维度关键词定义

分析表明多个维度的关键词集存在**显著重叠**:

| 重叠维度 | 共享关键词 | 维度核心关注 (focus) |
|---------|-----------|-------------------|
| D1 "战略链接与价值认知" | 战略, 目标, 业务, 价值, 支撑, 痛点 | 战略理解深度 |
| D6 "课题的战略价值" | 课题, 价值, 战略, 效能, 需求, 痛点, 组织 | 价值贡献度 |
| D7 "目标与规划的前瞻性" | 目标, 规划, 里程碑, 资源, 计划, 推进, 实施 | 落地可行性 |

**具体场景**: 句子 "本课题的战略价值在于通过解决业务痛点支撑组织目标达成"

这句话包含 D6 的关键词 (课题, 战略, 价值, 痛点) 和 D1 的关键词 (战略, 业务, 价值, 支撑, 目标)。在 `_build_evidence()` 中，D1 和 D6 都可能匹配到同一个句子。

**核心问题**: D1 "战略链接与价值认知" 评价的是汇报人是否能将自己的工作与公司战略有效链接 — 这是一个**个人认知维度**。但证据引用的是关于 "课题价值" 的句子，实际上属于 D6 的范畴。证据与维度含义脱节。

---

### 问题 3: 课程专用关键词加剧表面匹配

**涉及代码**: `rules.py` 第 305-467 行, `_build_action_learning_course_definition()` 函数

课程专用评分标准为维度注入了大量领域术语作为关键词。例如，"行动学习-认知升级" 课程的 "直面问题" 维度关键词:
```python
"直面问题": [
    "测评", "商业综合推理", "管理技能", "管理个性",
    "管理风格", "职业锚", "PEST", "短板", "瓶颈",
],
```

**具体场景**: 文档中包含 "我通过PEST分析识别了外部市场环境中的机会与威胁" 和 "我的主要短板是缺乏跨部门沟通经验" 两句。

算法为 "直面问题" 选取 evidence 时:
- 扫描到 PEST → 匹配第一句，但该句是环境分析工具的使用描述
- 扫描到 短板 → 匹配第二句，但该句是表面描述，未展现问题穿透深度

**核心问题**: "直面问题" 的评价核心是 "问题穿透深度" — 即学员是否能深入挖掘根因、直面核心矛盾。但关键词匹配到的句子可能只是**提到**了相关术语，并没有体现维度真正的评估意图。一个学生可以简单地用 "我做了PEST分析" 这类表面描述触发关键词匹配，但实际上并未展示深度分析能力。

### 问题 4: LLM 路径证据无接地验证

**涉及代码**: `live_scoring.py` 第 62-68 行和 第 207 行

LLM 系统提示要求:
```python
"Evidence must quote or paraphrase a short supporting excerpt from the provided material."
```

然后在后处理中仅做截断:
```python
evidence = _limit(str(item.get("evidence") or "").strip(), 80)
```

**核心缺陷**:
- **无接地验证**: 没有检查 LLM 生成的 evidence 是否**真实存在于**原始文档中
- **无归一性检查**: LLM 可以编造看起来合理的引用，系统无法察觉
- **截断破坏语义**: 80 字截断可能切断关键上下文，使引用失去意义
- **未使用知识库内容**: 知识库文件（如 `知识库和评分标准/1.md`）包含详细的评分指引（"重点关注：..."），但 prompt 没有明确要求 LLM 据此判断什么是好的证据

LLM 生成的 evidence 示例如（推测）: "该学员通过PEST分析识别了市场机遇" — 这句话可能概括了原文意思但并非原文，在启发式路径下完全不可验证。

---

### 问题 5: `compute_confidence` 与 evidence 质量的脱节

**涉及代码**: `nodes.py` 第 260-324 行

`compute_confidence` 函数的置信度判定标准:
```python
if is_llm and coverage >= 0.5 and text_length >= 1000:
    level = "high"
```

置信度仅基于**关键词覆盖率**和**文本长度**，与 evidence 的质量完全无关。一个维度可能:
- 有 high 置信度（因为关键词覆盖率高）
- 但 evidence 却是偏门信息（因为第一匹配偏见）
- 从而不会被标记为需要人工审核 (`human_review_gate` 仅标记 low 置信度的维度)

这意味着**即使 evidence 是偏门信息，只要关键词覆盖率达标，就被视为 "高质量评分"** 而跳过复审。

---

## 三、问题根因总结

| # | 根因 | 影响 |
|---|------|------|
| 1 | 证据选取使用第一匹配策略，无质量排序 | 边缘信息成为证据 |
| 2 | 关键词集无语义边界，跨维度碰撞 | 证据张冠李戴 |
| 3 | 关键词匹配≠维度核心含义匹配 | 表面术语触发成为证据 |
| 4 | LLM 路径无接地验证 | 无法保证证据真实存在 |
| 5 | 置信度与 evidence 质量解耦 | 偏门信息绕过人工审核 |
| 6 | `_build_comment` 纯模板化，与 evidence 无关 | comment 不引用证据内容 |

---

## 四、影响范围

| 模块 | 影响程度 | 说明 |
|------|---------|------|
| `scoring.py::_build_evidence()` | 严重 | 启发式路径的核心缺陷 |
| `scoring.py::score_submission()` | 严重 | 决定整个系统的证据质量 |
| `live_scoring.py::live_score_submission()` | 中等 | LLM 路径缺乏验证环节 |
| `rules.py` 关键词定义 | 中等 | 关键词设计可优化但不能根治 |
| `graph/nodes.py::compute_confidence()` | 轻度 | 置信度判定可考虑 evidence 质量 |
| `score_service.py` | 无 | 接口层不受影响 |
| 前端 API 契约 | 无 | evidence 仍是 ≤80 字字符串 |
